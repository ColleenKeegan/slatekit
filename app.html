App

The App component ( available in the slate.core project ) is an Application template that supports many common features needed for any type of application. Features include:

Features

Feature Description Example
1. Args Command line arguments support ( which can override any config settings ) -env=dev -log.level=info -region=ny
2. Envs Support and selection of multiple environments ( with defaults ) local, dev, qa, stg, pro
3. Logs Logging and error handling ( any log provider can be easily added ) console / file logger, custom logger
4. Config Configuration files for each environment ( with support for config inheritance ) env.conf, env.local.conf, env.qa.conf
5. Settings Settings can be in config, encrypted, or a file in user folder ( for added security ) ${user}/mycompany/myapp/database.conf
6. Encryption Encryption and decryption support Using AES
7. Life-cycle Application life-cycle events check(), init(), accept(), execute(), shutdown()
8. Context An application context to contain and provide access to many services encryptor, logger, settings, environment, etc

Usage

The App template is used as the base class for all Slate application types. This makes creation, setup, and execution of any app type consistent and very simple.

App Type Description Example
1. Console A simple console application template with diagnostics and logging Download the sample apps
2. CLI The CLI( Command Line Interface ) to your APIs Download the sample apps
3. Server The web server hosting your APIs Download the sample apps

Setup

You can create a new Application by subclassing AppProcess

Example


import slate.core.app.AppProcess

class SampleAppBatch extends AppProcess
{
  /**
   * Initialize your app here
   *
   * @return
  */
  override def onInit(): Unit =
  {
    // The base init can take care of some things such as:
    // 1. loading the "resources/env.conf" config file as "confBase"
    // 2. loading the selected environment "env" from either 
    //    the command line first ( -env="dev1:dev" ) or "env.conf"
    // 3. setting up "conf" inherited config from "env.dev.conf" + "env.conf"
    
    // Initialize the context with common app info
    ctx = new AppContext (
      env  = env,
      cfg  = conf,
      log  = new LoggerConsole(getLogLevel()),
      con  = conf.dbCon( None, Some(".slate") ).getOrElse(null),
      enc  = new Encryptor("wejklhviuxywehjk", "3214maslkdf03292"),
      inf  = buildAbout(),
      app  = new AppMeta(),
      host = Host.local(),
      lang = Lang.asScala()
    )
    // your code here.
  }


  /**
   * You implement this method to executes the app
   *
   * @return
   */
  override def onExecute():Result =
  {
    // simulate work
    info("simulating work for 2 seconds. please wait...")
    Thread.sleep(2000)

    success()
  }


  /**
   * HOOK for when app is shutting down
   */
  override def onShutdown(): Unit =
  {
    info("app shutting down")
  }


  /**
   * HOOK for adding items to the summary of data shown at the end of app execution
   * @param args
   */
  override def collectSummaryExtra(args:ListBuffer[String]): Unit =
  {
    args += (meta.about.name + ": extra 1  = extra summary data1")
    args += (meta.about.name + ": extra 2  = extra summary data2")
  }


  /**
   * build information about this application for diagnostics
   * @return
   */
  override def buildAbout():About = {
    // Get info about app from base config "env.conf" which is common to all environments.
    new About(
      name     = confBase.getString("app.name"     , "Sample App"),
      desc     = confBase.getString("app.desc"     , "Sample to show the base application"),
      region   = confBase.getString("app.region"   , "ny"),
      version  = confBase.getString("app.version"  , "0.9.1"),
      url      = confBase.getString("app.url"      , "http://sampleapp.slatekit.com"),
      group    = confBase.getString("app.group"    , "codehelix.co"),
      contact  = confBase.getString("app.contact"  , "kishore@codehelix.co"),
      tags     = confBase.getString("app.tags"     , "slate,shell,cli"),
      examples = confBase.getString("app.examples" , "")
    )
  }   
   
  // ...
}

Environments

You can setup different environments ( local, dev, qa, etc ) for your application. The following environments are available by default. Since environments need to have config files (see below) and some environments may share common config settings, there is the concept of a base or common environment which the other environments are based from.

Env Desc Parent
Common Common -
Loc Development ( local ) Common
Dev Development ( shared ) Common
QA Quality Assurance / Test Common
Stg Staging / Beta Common
Prod env.conf Common

Example

In code, an environment is represented as an EnvItem

import slate.common.{Env, EnvItem}

// Environment definition
val env = new EnvItem("dev1", Env.DEV, "dev1.dev")
Env.set(env)
Env.isDev
Env.isQa
Env.isUat
Env.isProd
Env.current
  1. You can create custom environments
  2. The most common environemnts ( local, dev, qa, stg, prod ) are available by default
  3. Other typical environments are ( local, dev1, dev2, qa1, qa2, stg1, stg2, prod ) to support current version + production version in each environment

Configuration

All the different types of Apps and Sample Apps using a common configuration scheme. For each app, there is a single common configuration file env.conf and then 1 environment specific configuration file for each environment. e.g. env.dev.conf All the environment configs are located in the resources directory. Slate Kit supports inherited configs, so you can setup your env.dev.conf to inherit from env.conf.

Env File Parent Desc
Common /resources/env.conf - Common config
Loc /resources/env.loc.conf env.conf Development ( local )
Dev /resources/env.dev.conf env.conf Development ( shared )
QA /resources/env.qa1.conf env.conf Quality Assurance / Test
Stg /resources/env.stg.conf env.conf Staging / Beta
Prod /resources/env.pro.conf env.conf Production / Live

Example 1: Common config ( env.conf )

You can keep all the settings that are common to all environments in this common config. Some examples of common settings are application info. 


# environment selection
# this can be overriden on the commandline via -env=qa
env.name = "loc"
env.mode = "dev"

# This config file is :
# 1. common to all other config files
# 2. inherited by all other config files.
# 3. settings can be overriden in the derived files.
# these can be overriden in each of the environment specific config files
# e.g.
# dev = env.dev.conf ( development )
# qa  = env.qa1.conf ( qa testing  )
# loc = env.loc.conf ( local dev   )
# pro = env.pro.conf ( production  )
app.id      = "slate.examples"
app.company = "codehelix"
app.name    = "Sample App"
app.desc    = "Sample console application to show the base application features"
app.region  = "ny"
app.version = "0.9.1"
app.url     = "http://sampleapp.slatekit.com"
app.group   = "codehelix.co"
app.contact = "kishore@codehelix.co"
app.tags    = "slate,shell,cli"
app.examples = "sampleapp -env=dev -log.level=debug -region='ny' -enc=false"

# log and level
log.name    = "@{app.name}-@{env.name}-@{date}.log"
log.enabled = true
log.level   = "info"

# DB Settings - defaulted to dev database
db.enabled     = false
db.source      = "conf"
db.driver      = "com.mysql.jdbc.Driver"
db.url         = "jdbc:mysql://localhost/db1"
db.user        = "root"
db.pswd        = "123456789"

Example 2: Environment specific ( env.qa.conf )

You can override any settings from common config in the environment specific configs. For example, you can override database settings for QA and make them encrypted

# Example: Override the db settings for this environment
# and use encrypted values.
db.enabled     = true
db.source      = "conf"
db.driver      = "com.mysql.jdbc.Driver"
db.url         = "jdbc:mysql://localhost/QA1"
db.user        = "7g7bENenJnO2RCZ9iBnBZw"
db.pswd        = "8r4AbhQyvlzSeWnKsamowA"

Example 3: Reference Config file in user directory.

For added security, you can extract some settings into a separate config file that is kept in the user directory. You can create separate config files for different settings such as database, AWS, etc. See the settings section for more info


# Example: Override the db settings for prod to indicate
# loading the info from a file called "db-pro.txt" from the users folder.
# e.g. /<user>/<company>/<app>/db.pro.txt
#
# This is an added security measure to avoid production settings
# being stored in the config directory/files. User folder is :
#
# 1. more secure
# 2. not stored in source control
# 3. stored in /<user>/<company>/<app>/db.pro.txt
#
# NOTE: This is a pattern recommended by amazon.
db.enabled = true
db.source  = "user.file"
db.file    = "db-pr1.txt"

Example 4: User directory (extract settings into separate file in user directory )

Settings can be simply copied pasted into a separate config file and stored in the user folder. e.g. db.pro.txt


db.driver = "com.mysql.jdbc.Driver"
db.url    = "jdbc:mysql://localhost/test1"
db.user   = "root"
db.pswd   = "123456789"
  1. On windows the user directory is typically c:/users/{username}
  2. The folder path for the file in user directory should be: {username}/{company}/{app}/{filename}
  3. Example file is: kreddy/.codehelix/sampleapp1/database.conf

Settings

There are configuration settings available for different components. For example, database settings, logging settings, etc. These settings can be setup in either:

  1. inside common config env.conf - shared by all environments.
  2. inside environment config env.dev.conf - specific to environment and overrides common config
  3. external config file in user dir kreddy/mycompany/app1/log.conf

Area Prefix Example Location Desc
Env env env.name env.conf Environment selection ( dev, qa1, stg, pro )
Database db db.url env conf or userdir:db-dev.conf Database settings including, driver, db name, user, password
Logging log log.level env conf or userdir:log.conf The logging configuration, log level, enabled, file name
Encryption enc enc.enabled env conf or userdir:log.conf The logging configuration, log level, enabled, file name
AWS aws aws.key env conf or userdir:aws.conf AWS configuration including the key, password
  1. It is recommended to encrypt settings ( see the config component in utils for more info )
  2. It is also recommended to store Database, AWS and other sensitive settings in separate files in the user directory

Arguments

You can setup command line args for your application by building a argument schema.


class SampleAppBatch extends AppProcess 
{

  // STEP 1. setup the command line arguments.
  // NOTE:
  // 1. These values can can be setup in the application.conf file
  // 2. If supplied on command line, they override the values in .conf file
  // 3. If any of these are required and not supplied, then an error is display and program exists
  // 4. Help text can be easily built from this schema.
  argsSchema.addText("env"   , "the environment to run in", true , "dev"  , "dev"  , "dev1|qa1|stg1|pro" )
            .addText("log"   , "the log level for logging", false, "info" , "info" , "debug|info|warn|error")
            .addFlag("enc"   , "whether encryption is on" , false, "false", "false", "true|false")
            .addText("region", "the region linked to app" , false, "us"   , "us"   , "us|europe|india|*")


	// ...
}
  1. The syntax for command line args is {app} -name=value
  2. Example is myapp -env=dev -log.level=info -region="ny"

Startup

All the different types of applications ( Console, CLI, Server - Web API ) all start up in the same way. This is accomplished by extending from the Slate Kit AppProcess class. Because most apps will need to have a way to initialize the environment, configuration, logging and more, we have templated this process to make it easier while also allowing for easy customization. The following are the series of actions that occur during startup.

Step Reference Location Desc
1. Configure command line args argsSchema = new ArgsSchema()... Your application
see sample app		 Configure the command line args( required | optional ) needed by your application.
2. Load common config env.conf AppProcess.scala env
3. Check command line args -env=dev -log.level=debug n/a The command line args provided are validated against your args schema from step above.
4. Load environment -env=dev command line or env.conf The environment selection is done by supplying the environment in the command line or by in the env.conf. The command line value overrides the value from the env.conf.
5. Load environment config env.dev.conf | env.qa1.conf AppProcess.scala The environment specific config is loaded based on the environment selection from step above
6. Load context ctx = new AppContext(...) Your application The AppContext must be created which holds many dependent common componens / services ( e.g. logger, conf, env, encryptor, etc ). This context is available
7. Load other data env.name env env

Life-cycle

The application has 4-5 life-cycle events you can hook into.

Step Method Desc
1. Check check() Checks the command line arguments
2. Initialize init() Initialize your application
3. Accept accept() Post initialization
4. Execute execute() Execute the code for your application.
5. Shutdown shutDown() Shutdowns your application

App Runner

The AppRunner class is a convenience class for executing your application. It calls the life-cycle events in the appropriate order


package slate.core.app

import slate.common.{Result}

object AppRunner
{
  /**
   * runs the app with the arguments supplied.
   * @param app
   * @param args
   * @return
   */
  def run(app: AppProcess, args:Option[Array[String]]): Result =
  {
    // 1. Check the command line args
    val result = app.check(args)
    if(!result.success ){
      return result
    }

    // 2. Request for help
    // e.g.
    // - ?
    // -about
    if(result.isExit || result.isHelpRequest){
      return result
    }

    // 3. Allow app to initialize
    app.init()

    // 4. Accept the initialize
    // NOTE: This serves as a hook for post initialization
    app.accept()

    // 5. Execute the app
    val res = app.exec()

    // 6. Shutdown the app
    app.shutdown()

    // 7. Return the result from execution
    res
  }
}
